# Global CDN Integration Guide

This guide documents the architecture, setup instructions, deployment configuration, cost-tracking strategies, and performance measurement practices for serving NexaSphere static assets via a global Content Delivery Network (CDN) utilizing AWS S3 and AWS CloudFront.

---

## 1. Architecture Overview

```
                        +-----------------------+
                        |     User Browser      |
                        +-----------+-----------+
                                    |
                                    | Requests index.html, JS, CSS
                                    v
                        +-----------+-----------+
                        |  CloudFront Edge CDN  |
                        +-----------+-----------+
                                    |
            +-----------------------+-----------------------+
            | (Cache Miss)                                  | (Cache Hit)
            v                                               v
+-----------+-----------+                       +-----------+-----------+
|    AWS S3 Origin      |                       |   Served Directly     |
| (Static Files Bucket) |                       |     from Cache        |
+-----------------------+                       +-----------------------+
```

1. **Client Request:** The user browser requests static assets.
2. **CloudFront CDN Cache:** If the asset is cached globally at a nearby Edge Location (Cache Hit), it is served instantly.
3. **AWS S3 Origin:** If the asset is not cached (Cache Miss), CloudFront fetches it securely from the S3 bucket using Origin Access Control (OAC), serves the user, and caches the asset.

## 2. AWS Setup Instructions

### Step 1: AWS S3 Bucket Setup

1. Open the [AWS S3 Console](https://s3.console.aws.amazon.com/).
2. Create a bucket named `nexasphere-static-assets` (or similar).
3. **Block Public Access:** Keep "Block _all_ public access" **enabled** (CloudFront will access files securely using OAC).
4. Enable **Bucket Versioning** for easy rollback of files if needed.

### Step 2: AWS CloudFront Distribution Setup

1. Open the [AWS CloudFront Console](https://console.aws.amazon.com/cloudfront/).
2. Click **Create Distribution**.
3. **Origin Domain:** Select the S3 bucket created in Step 1.
4. **Origin Access:** Select **Origin access control settings (recommended)**. Click **Create control setting** and use the default settings (Sign requests).
5. **Viewer Protocol Policy:** Choose **Redirect HTTP to HTTPS**.
6. **Allowed HTTP Methods:** Select `GET, HEAD`.
7. **Cache Key and Origin Requests:** Select **CachePolicyAndOriginRequestPolicy** and use the **CachingOptimized** policy.
8. Click **Create Distribution**.
9. **Update S3 Bucket Policy:** Copy the policy generated by CloudFront, go back to the S3 bucket permissions settings, and paste it into the S3 bucket policy editor. This allows CloudFront to read S3 objects securely.

### Step 3: Configure CORS on S3 Bucket

Under S3 Bucket Permissions, add the following CORS policy to allow cross-origin requests for fonts, JSON metadata, and other files:

```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["GET", "HEAD"],
    "AllowedOrigins": ["https://your-domain.vercel.app", "http://localhost:5175"],
    "ExposeHeaders": [],
    "MaxAgeSeconds": 3000
  }
]
```

## 3. Caching and Deployment Invalidation Policy

### Caching Strategy

Static assets are split into two categories to maximize loading speeds and prevent stale code issues:

1. **Immutable Assets (Hashed JS, CSS, Images):**
   - **Path:** `/assets/*`
   - **Cache-Control Header:** `public, max-age=31536000, immutable`
   - **Why:** Vite automatically appends content hashes (e.g. `index-a4b89c.js`) to filenames when building. These files never change; if code updates, Vite generates a new filename. We cache them permanently.

2. **Mutable Entrypoints (index.html, sw.js, manifest.webmanifest):**
   - **Path:** `/index.html`, `/sw.js`, `/manifest.webmanifest`, `/favicon.ico`
   - **Cache-Control Header:** `public, max-age=0, must-revalidate`
   - **Why:** The entrypoint HTML and service worker control app logic updates. They must be validated against the server immediately to determine if a newer version of the site has been deployed.

### Invalidation on Deploy

To ensure immediate propagation of code changes on deploy:

1. When assets are deployed, a CloudFront cache invalidation is triggered.
2. In the deployment script `scripts/deploy-cdn.sh`, we run:

   ```bash
   aws cloudfront create-invalidation --distribution-id <id> --paths "/*"
   ```

3. This invalidates the cached entrypoint files at all edge locations, forcing them to re-verify files on the origin (S3) next time they are requested.

## 4. Cost Tracking and Optimization

Serving assets via CDN can be highly cost-effective if configured correctly. Implement the following guidelines:

1. **CloudFront Price Class:**
   - **Recommendation:** Use **Price Class 100** (US, Canada, Europe edge locations only) or **Price Class 200** (excludes most expensive locations) to minimize egress charges if your primary community base is regional.
2. **Set Up AWS Budgets:**
   - Go to [AWS Billing Console -> Budgets](https://console.aws.amazon.com/billing/home#/budgets).
   - Create a budget with a $5.00 limit (or higher depending on your traffic scale) and configure an email alert if forecasted or actual spend exceeds 80%.
3. **S3 Storage Lifecycle Rules:**
   - Configure a lifecycle rule in the S3 bucket to transition old unreferenced asset versions to **Glacier Deep Archive** or delete them automatically after 30 days to save storage costs.
4. **CloudFront Reports:**
   - Monitor the **CloudFront Usage Reports** and **Cache Statistics** monthly to track bytes transferred and average latency.

## 5. Performance Improvement Measurement

To evaluate the success and speed improvements of the CDN integration:

1. **Lighthouse Audits:**
   - Measure **Largest Contentful Paint (LCP)**, **First Contentful Paint (FCP)**, and **Cumulative Layout Shift (CLS)** before and after CDN setup.
   - Target LCP: `< 2.5 seconds`.
2. **CloudFront Cache Hit Ratio:**
   - Check the Cache Hit Ratio in CloudFront Console -> Cache Statistics.
   - **Target:** `> 90%` for asset requests. If the hit ratio is lower, audit if assets are missing content hashes or if cache headers are misconfigured.
3. **Core Web Vitals Monitoring:**
   - Track Real User Monitoring (RUM) latency using services like Sentry Web Vitals or Google Search Console Core Web Vitals report.
